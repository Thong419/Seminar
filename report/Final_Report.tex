\section{Executive Summary}
This project delivers an end-to-end natural language processing system for fake news and misinformation detection with production-oriented design choices rather than a notebook-only prototype. The final system combines a classical baseline, a transformer classifier, an agentic orchestration layer, evidence retrieval, explainability, privacy protection, monitoring, and a deployable user interface. The implementation directly addresses the rubric in Prokct-2 by covering business framing, data handling, model development, evaluation, deployment, responsible AI, and agentic reasoning.

The deployed inference stack exposes a FastAPI backend and a Streamlit frontend. The backend provides prediction, analysis, health, version, and agent endpoints; the frontend renders predictions, evidence cards, trust signals, and execution traces. The core classifier is a fine-tuned RoBERTa model, while a TF--IDF plus Logistic Regression baseline serves as a reproducible comparison point. The system also includes Optuna-based hyperparameter tuning, privacy masking before inference, drift and confidence monitoring, and a dedicated UNCERTAIN review state for evidence-conflict or low-confidence cases.

The key engineering goal of the project is not only strong classification performance but also trustworthy behavior in demonstration and operational settings. For that reason, the final design treats model confidence as only one signal among several. External evidence, source credibility, evidence quality, contradiction detection, and trace capture all contribute to the final user-facing result. This allows the system to explain why a claim was labeled REAL, FAKE, or UNCERTAIN and makes the workflow appropriate for academic defense and product-style review.

\section{Problem Definition}
Fake news detection is a high-value NLP problem because misinformation spreads faster than manual verification processes can scale. Moderation teams, fact-checking groups, and internal trust-and-safety operations need tools that can triage content, reduce review time, and surface signals for human judgment. An automated system has practical value if it can support reviewers with fast, consistent, and explainable recommendations rather than opaque labels.

In this project, the target stakeholders are moderators, newsroom verification staff, and platform operators who need an evidence-aware first-pass assessment of article-like text. The business value comes from reduced manual effort, improved response time, and better prioritization of high-risk content. The technical objective is to build a binary classification system with strong accuracy and F1 while preserving deployment readiness and interpretability. The operational objective is to expose that model in a user-facing system that remains safe under imperfect evidence conditions.

NLP is required because the input is free-form text whose meaning depends on word choice, semantic context, and longer-range dependencies. Traditional heuristics alone are not sufficient for robust classification, especially for ambiguous or adversarially phrased claims. A transformer model such as RoBERTa can capture these linguistic dependencies more effectively than a bag-of-words model, while retrieval and explainability components provide external grounding and human-readable output.

The main success metrics are technical and operational. Technical metrics include accuracy, precision, recall, F1, and response latency. Operationally, the system should provide a usable explanation, retrieve relevant evidence when available, and signal uncertainty when classifier and evidence do not align. Those requirements shaped the final architecture more strongly than raw benchmark performance alone.

\section{Literature Review}
Fake news detection has been studied from multiple directions, including lexical feature engineering, social-context modeling, graph methods, and transformer-based classification. Earlier practical systems commonly used TF--IDF features with linear classifiers because those methods are efficient, interpretable, and easy to deploy. They remain important as baselines because they reveal whether a dataset and preprocessing pipeline are already sufficient to solve a simplified version of the task.

Recent progress in transformer language models has shifted the state of practice toward contextual representation learning. BERT demonstrated the value of deep bidirectional pretraining for downstream classification tasks, and RoBERTa improved on that recipe through larger-scale and more robust pretraining choices. These models are especially suitable for misinformation detection because deceptive claims often depend on context, framing, negation, and semantic composition rather than isolated keywords.

A second relevant line of work concerns explanation and trust. High-accuracy models are not enough in domains where a user may question the basis of a prediction. Token-level attribution, evidence retrieval, and conflict detection are increasingly used to make model outputs reviewable. This project follows that direction by combining model prediction with evidence-aware reasoning instead of presenting confidence alone as the final answer.

A third line of work concerns production deployment and continual monitoring. Many academic NLP systems stop at offline evaluation, but production-style systems must consider API behavior, privacy, drift, observability, and model lifecycle. The course rubric explicitly requires those aspects, so this project extends beyond training into deployment architecture, monitoring artifacts, and conceptual retraining strategy.

\section{Dataset Description}
The repository configuration defines a local dataset source at \texttt{data/raw/fakenewsnet.csv} with fields \texttt{text} and \texttt{label}. The dataset configuration uses binary labels \texttt{fake} and \texttt{real}, which aligns with both the baseline and transformer classification pipelines. The final system assumes article-level input and binary decision output, with later evidence-aware logic supporting an operational UNCERTAIN review state.

The data configuration specifies a 70/15/15 train, validation, and test split with random seed 42 and stratification enabled. This is implemented in the repository rather than stated only in documentation, which supports reproducibility. Stratification is particularly important in fake news detection because skewed label distributions can inflate naive accuracy while harming minority-class behavior.

The repository includes structured data handling code under \texttt{src/data/}, including dataset preparation, preprocessing, validation, and splitting modules. This means data management is not hidden inside notebooks but encoded as reusable software components. That supports the course requirement for professional structure and makes it easier to rerun experiments or audit assumptions.

The main dataset limitations are typical for fake news corpora: source-domain bias, topic imbalance, and the possibility that label quality depends on historical fact-checking context rather than intrinsic linguistic quality alone. Those limitations matter because a strong classifier may still generalize poorly when content style or topic mix changes. For that reason, the final system includes monitoring and a dedicated robustness discussion instead of claiming permanent or universal reliability.

\section{Data Processing}
The preprocessing pipeline is implemented in \texttt{src/data/preprocessing.py} and applies multiple deterministic cleaning steps: HTML removal, URL removal, lowercasing, special-character handling, and whitespace normalization. The design choice here is pragmatic. The baseline model benefits strongly from normalization because TF--IDF features are sensitive to superficial noise, while the transformer still receives cleaner input without losing the semantic structure of the article.

The cleaning pipeline is applied through typed configuration rather than hard-coded preprocessing behavior. This is important because it allows preprocessing decisions to remain explicit and reproducible. The code also drops empty texts, normalizes labels, and removes duplicate rows by text-label pair, which reduces evaluation distortion due to repeated examples.

The split pipeline is implemented in \texttt{src/data/splitter.py}. It first validates that split ratios sum to one, then creates a held-out test set and a train/validation split inside the remaining data. Stratification is used when the label distribution supports it. This two-stage design is simple, auditable, and appropriate for a course project that must emphasize correct experimental handling.

A production-minded system also needs to consider missing, noisy, and potentially biased data. The code already guards against missing labels and empty text after preprocessing. At a higher level, the project treats robustness and bias as explicit documentation and monitoring concerns because a clean split is necessary but not sufficient for trustworthy deployment.

\section{Baseline Model (TF--IDF + Logistic Regression)}
The baseline model is configured in \texttt{configs/training.yaml} and trained through \texttt{src/training/train\_baseline.py}. It uses TF--IDF with up to 20,000 features, unigrams and bigrams, document-frequency thresholds, and sublinear term frequency. Logistic Regression uses a balanced class weighting strategy and a solver suitable for sparse linear classification.

This baseline is not included merely for formality. It performs two important roles. First, it provides a transparent benchmark that verifies the quality of the dataset and preprocessing pipeline. Second, it provides a concrete comparison for the course requirement that the final model be evaluated against at least one simpler alternative.

The recorded evaluation metrics show that the baseline is already very strong. Its accuracy is approximately 0.9913 and its F1 is approximately 0.9904. This means the dataset carries strong lexical and phrase-level cues that a linear classifier can exploit effectively. That result strengthens the final report because it shows the transformer must outperform a non-trivial benchmark rather than an intentionally weak strawman.

Despite its strengths, the baseline has known limitations. It cannot model long-range compositional meaning as effectively as a transformer, and it is more brittle under paraphrase or contextual framing changes. Those constraints justify the transition to RoBERTa while still preserving the baseline as a useful operational and scientific reference point.

\section{Transformer Model (RoBERTa)}
The main model is \texttt{roberta-base}, configured in \texttt{configs/model.yaml} and trained through \texttt{src/training/train\_transformer.py}. The training setup uses maximum sequence length 512, batch size 16, learning rate $2 \times 10^{-5}$, three epochs, weight decay 0.01, warmup ratio 0.1, and selection of the best checkpoint by validation F1. These choices are consistent with standard fine-tuning practice for transformer text classification.

The predictor used for inference is implemented in \texttt{src/inference/predictor.py}. It loads the tokenizer and sequence-classification model from the saved model directory and exposes a compact \texttt{predict()} interface returning label and confidence. This separation between training and inference is important for deployment, because the backend imports a predictor abstraction instead of embedding training logic into the serving path.

The transformer training entry point includes additional engineering behavior that reflects real-world constraints. It can discover the latest checkpoint, resume interrupted training, print diagnostics for environment debugging, and export comparison artifacts against the baseline. These are practical details that matter when moving from an experiment to a repeatable system.

RoBERTa is justified not only by its final score but by its ability to capture richer context. Fake news claims often depend on tone, negation, syntactic framing, or combinations of entities and events that are poorly represented by sparse linear features. The model therefore fits the problem better when the goal is robust deployed inference rather than only leaderboard performance.

\section{Agentic AI Architecture}
The agentic component is centered on \texttt{src/agent/controller.py}, which orchestrates classification, evidence retrieval, decision synthesis, explainability generation, metrics recording, and trace persistence. This directly satisfies the course requirement that the project contain multi-step reasoning and decision-making based on intermediate outputs. The workflow is not a static single-model call; it is a sequence of tools with explicit intermediate state.

The agent controller first classifies the article, then retrieves evidence, then combines those signals in a decision tool, and finally requests an explanation from the explainability service. Every step can be recorded in an execution trace. The final user-visible output contains not only prediction and confidence but also trust score, human review state, conflict flag, evidence summary, source information, and trace data.

The architecture is intentionally lightweight. Instead of using a large external orchestration framework for every action, the project implements a clear, testable controller around internally defined tools. This keeps the system maintainable while still exhibiting genuine agentic behavior. The design also respects the stated project constraint of not retraining or replacing the underlying RoBERTa checkpoint during the Day 1 to Day 4 evolution.

\section{Evidence Retrieval Pipeline}
The evidence retrieval pipeline is implemented through \texttt{src/agent/evidence\_tool.py} and the retrieval modules under \texttt{src/retrieval/} and \texttt{src/agents/retrieval/}. The tool extracts a claim, generates or reuses search queries, collects evidence sources, scores alignment, estimates source credibility, and summarizes whether the evidence supports or contradicts the article.

A notable design choice is the use of HTML-based fallback retrieval behavior instead of depending exclusively on a single external API key. The retrieval path can use a hybrid search client with Wikipedia and Bing HTML search clients, which improves robustness for academic demonstration. This matters because deployment readiness is not only about model serving; it also depends on whether the system can degrade gracefully when an external dependency is unavailable.

The pipeline assigns per-source fields including title, source, URL, relevance, provider, source credibility, stance, and matched terms. Those fields are then aggregated into support score, contradiction score, evidence quality score, and conflict flag. This makes the evidence layer computationally simple but operationally useful, because downstream components can reason over structured evidence rather than raw snippets only.

The current evidence scoring is heuristic, and the project explicitly documents that limitation. That is acceptable within the course scope because the requirement is to include tool usage and decision-making based on intermediate outputs, not to solve every retrieval or stance-analysis problem optimally. The important achievement is that the final system can operationalize evidence as a first-class signal.

\section{Explainability System}
The explainability system combines prediction context, evidence context, and token importance into a final human-readable explanation. It is called from the agent controller after the decision phase and returns both explanation text and important tokens. The frontend then renders these outputs together with trust score and evidence cards.

A key architectural correction made during the final stabilization was to treat the trust score coming from the decision layer as the authoritative value rather than recomputing it inside the explanation formatter. This avoids inconsistency between the decision module and the user-facing explanation. In product terms, this matters because conflicting trust values would reduce user confidence in the system more than a slightly lower raw score would.

The final explanation also includes evidence support, contradiction, source credibility, and human review state. As a result, the explanation is not limited to word attribution. It communicates the reasoning context around the prediction, which is much closer to how a reviewer would want to understand a suspicious article. This design therefore supports the rubric's requirement for explainability considerations for non-technical stakeholders.

The explainability layer remains intentionally lightweight and rule-compatible with the existing model. It does not require retraining the checkpoint and therefore respects the project constraint that the classifier remain unchanged while orchestration and presentation improve around it.

\section{Evaluation Methodology}
Evaluation is performed on a held-out test split created through the data splitting pipeline. The report uses accuracy, precision, recall, and F1 as primary classification metrics. These are complemented by confusion matrix inspection and error analysis artifacts stored in \texttt{artifacts/evaluation/}.

The methodology also includes direct baseline comparison. This is important because a high transformer score alone does not demonstrate good experimental practice. The course rubric requires at least one baseline and a discussion of why the final model is preferable. By training and evaluating both approaches on the same split logic, the repository satisfies that requirement in a reproducible way.

For the agentic layer, evaluation is broader than plain classification metrics. The project tests evidence retrieval, conflict handling, the UNCERTAIN state, trace serialization, and endpoint behavior. The agent endpoint tests in \texttt{tests/api/test\_agent\_endpoint.py} verify real, fake, and uncertain outcomes and ensure trace data is returned. This is an important methodological point because user trust depends on orchestration behavior, not only on classifier output.

Finally, operational methodology includes health and version endpoints, logging, monitoring artifacts, and container configuration. These elements do not replace classification evaluation, but they do satisfy the deployment and lifecycle requirements of the assignment.

\section{Deployment Architecture}

The Dockerfile builds a slim Python container with all application dependencies, while \texttt{docker-compose.yml} defines separate services for the API and frontend with health checks.

The backend is structured around \texttt{src/api/main.py}, \texttt{src/api/routes.py}, and \texttt{src/api/health.py}. Middleware records request IDs and latency, routes validate input, and the health and version endpoints support deployment probes. The frontend connects to the backend over HTTP and renders results in a way suitable for demonstration, screenshots, and defense discussion.

\section{Ethics, Privacy, and Limitations}
The system addresses privacy by masking personally identifiable information before inference and logging. The masking utility replaces detected entities such as emails, phone numbers, URLs, and addresses with stable placeholders. This design reduces the risk of storing raw sensitive text in downstream systems and satisfies the rubric's requirement for privacy analysis and minimization strategies.

Ethically, fake news detection systems carry risks of false positives, false negatives, and trust overreach. A false positive can suppress legitimate information, while a false negative can allow harmful misinformation to spread. The project mitigates this by introducing an operational UNCERTAIN state, conflict flags, and evidence-based explanation rather than presenting every output as a fully certain verdict.

Robustness remains a significant limitation. Evidence retrieval depends on public web availability, stance analysis is heuristic, and domain shifts could degrade both model and retrieval performance. The project does not hide these limitations; instead, it records them in documentation, monitoring design, and defense materials. This is an important part of responsible AI practice.

Another limitation is that the current monitoring stack is file-based and academic-demo oriented. It supports drift and confidence tracking, but it is not a full observability platform. Likewise, the current system does not perform automated retraining in production. These limitations are acceptable within scope because the rubric allows conceptual continual learning design even when a full implementation is not feasible.

\section{Future Work}
The highest-value technical improvement is stronger evidence alignment. A future version should replace heuristic support/contradiction detection with a dedicated stance model or retrieval-augmented reasoning pipeline. This would improve the quality of the trust score and reduce the number of cases where evidence is retrieved but poorly interpreted.

A second improvement is broader robustness and domain coverage. The classifier is strong on the evaluated distribution, but future work should evaluate out-of-domain generalization more aggressively, including multilingual content, platform-native short claims, and adversarial paraphrases. Additional fairness and subgroup analysis would also improve the responsible AI profile of the system.

A third improvement is operational hardening. The current Docker-based deployment is sufficient for grading and demonstration, but public deployment would benefit from centralized logs, rate limiting, managed secrets, CI-based image publishing, and cloud-native scaling policy. These are natural next steps for moving from academic prototype to real-world service.

Finally, the frontend could evolve from a demonstration UI into a reviewer workflow dashboard with case history, saved traces, and reviewer feedback collection. That would also strengthen the continual learning loop by creating an explicit path for new labeled corrections.

\section{Conclusion}
This project satisfies the course objective of building a realistic end-to-end NLP system rather than an isolated modeling experiment. It includes data handling, a baseline model, a transformer model, a deployable backend and frontend, an agentic orchestration layer, monitoring concepts, privacy handling, and defense-ready documentation. The result is both technically strong and architecturally coherent.

RoBERTa provides the best classification performance, but the most important final submission contribution is the surrounding system design. Evidence retrieval, trust scoring, explanation, conflict detection, trace capture, and operational endpoints transform the classifier into a more reviewable decision-support tool. This is what makes the project aligned with production-oriented NLP practice.

The final verdict is therefore not only that the model performs well, but that the repository now demonstrates the full lifecycle expected by Prokct-2: problem framing, structured implementation, evaluation, deployment, agentic behavior, monitoring, and responsible AI reflection.

\section{Defense Demo Script Summary}
The defense demonstration should highlight the system as an integrated product rather than a sequence of disconnected files. A strong demo path is therefore: start from the Streamlit UI, show a factual article example, show a suspicious or fabricated example, then explain how evidence, trust score, and trace differ between those cases.

An effective demo should include at least one scenario where the classifier and evidence align and one where the final system escalates to UNCERTAIN. That contrast is critical because it shows why the agentic layer exists. Without it, the defense risks collapsing into a generic text-classification demonstration.

The demo should also verify two operational endpoints live: \texttt{/health} and \texttt{/version}. This is useful because it immediately communicates deployment readiness and shows that the system is not only a static UI mockup. Finally, if time allows, the examiner should be shown the trace or evidence cards for at least one example to reinforce transparency claims.
